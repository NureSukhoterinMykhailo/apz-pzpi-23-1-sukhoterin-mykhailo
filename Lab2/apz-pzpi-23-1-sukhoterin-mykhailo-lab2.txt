1 МЕТА РОБОТИ
Метою лабораторної роботи є розробка мобільного застосунку для платформи Android мовою Kotlin, який є частиною програмної системи «LifeBalance». Застосунок має забезпечувати взаємодію користувача із системою для авторизації, перегляду дашборду, керування звичками, цілями та аналітикою, а також демонструвати можливість синхронізації даних із серверною частиною через REST API.
2 ХІД РОБОТИ
У ході виконання роботи було реалізовано клієнтську частину системи LifeBalance для мобільних пристроїв. Мобільний застосунок виступає окремим клієнтом системи, який взаємодіє з backend-рівнем та надає користувачу швидкий доступ до ключових функцій моніторингу балансу життя.
2.1 Опис інженерних рішень
Для реалізації мобільного застосунку було обрано платформу Android та мову Kotlin. Таке рішення відповідає вимогам лабораторної роботи та дозволяє створити сучасний мобільний застосунок із підтримкою Android SDK, AndroidX та Material Design.
Архітектура застосунку орієнтована на розділення інтерфейсу користувача, логіки обробки даних та взаємодії із зовнішніми сервісами. У межах прототипу основна логіка розміщена у MainActivity.kt, де реалізовано переходи між екранами авторизації, дашборду, звичок, цілей, аналітики та профілю користувача.
Застосунок передбачає подальшу інтеграцію із серверною частиною через REST API. Через API можуть передаватися дані про звички, цілі, прогрес користувача, аналітичні показники та інформація, отримана від IoT- або wearable-пристроїв.
Для роботи проєкту використано JDK 17, Gradle, Android Gradle Plugin, AndroidX та Kotlin-плагін. У налаштуваннях Gradle увімкнено підтримку AndroidX, оскільки застосунок використовує сучасні Android-бібліотеки.
2.2 Опис будови мобільного застосунку
Мобільний застосунок складається з таких логічних частин:
•	екран авторизації користувача;
•	Dashboard для перегляду загального стану балансу життя;
•	модуль керування звичками;
•	модуль керування цілями;
•	модуль аналітики;
•	профіль користувача та демонстраційна синхронізація з wearable-пристроєм;
•	рівень майбутньої взаємодії з backend API.
На екрані авторизації користувач вводить email та пароль або може обрати демонстраційну дію входу через Google OAuth. Після входу відкривається головний екран Dashboard, на якому відображаються кількість виконаних звичок, активні цілі та загальний показник балансу.
2.3 Взаємодія з іншими частинами системи
LifeBalance складається з мобільного застосунку, web-клієнта, серверної частини та бази даних. Мобільний застосунок взаємодіє з сервером за допомогою REST API. Серверна частина відповідає за збереження та обробку даних користувача, синхронізацію між web- та mobile-клієнтами, формування аналітики та обробку повідомлень.
У майбутньому застосунок може отримувати дані з wearable-пристроїв, наприклад інформацію про активність, сон, кількість кроків або інші показники. Ці дані не обробляються напряму як медичні, а використовуються для покращення аналітики та рекомендацій щодо балансу життя.

3 ПРОГРАМНА РЕАЛІЗАЦІЯ
Нижче наведено фрагмент програмної реалізації мобільного застосунку LifeBalance мовою Kotlin. У фрагменті показано створення екрана авторизації та перехід до головного екрана Dashboard.
// file: MainActivity.kt
class MainActivity : Activity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        showLogin()
    }

    private fun showLogin() {
        val layout = baseLayout("LifeBalance")
        layout.addView(text("Balance your habits, goals and lifestyle", 16f, true))

        val email = input("Email", InputType.TYPE_TEXT_VARIATION_EMAIL_ADDRESS)
        val password = input("Password", InputType.TYPE_CLASS_TEXT or InputType.TYPE_TEXT_VARIATION_PASSWORD)

        layout.addView(email)
        layout.addView(password)
        layout.addView(button("Login") { showDashboard() })
        layout.addView(button("Continue with Google OAuth") {
            toast("Google OAuth demo action")
        })
    }

    private fun showDashboard() {
        val layout = baseLayout("Dashboard")
        layout.addView(card(
            "Today",
            "Completed habits: ${habits.count { it.completed }}/${habits.size}
" +
            "Active goals: ${goals.size}
" +
            "Lifestyle balance score: 76%"
        ))
    }
}

4 ПЕРЕВІРКА ВІДПОВІДНОСТІ ВИМОГАМ
Звіт містить опис прийнятих інженерних рішень, будову мобільного застосунку та його взаємодію з іншими частинами системи. Також у звіт включено UML діаграму прецедентів, UML діаграму компонентів та дві додаткові діаграми на вибір: ER-модель даних і UML діаграму діяльності.
Код мобільного застосунку реалізовано для платформи Android мовою Kotlin. Застосунок є частиною системи LifeBalance, має власний інтерфейс, екрани авторизації, дашборду, звичок, цілей, аналітики та профілю, а також передбачає подальшу взаємодію із серверною частиною через REST API.
5 ВИСНОВКИ
У результаті виконання лабораторної роботи №2 було розроблено мобільний застосунок системи LifeBalance на базі ОС Android. Було реалізовано базову клієнтську частину системи, створено інтерфейс користувача, описано архітектуру застосунку та показано його взаємодію з backend-частиною.
Створені UML-діаграми підтверджують відповідність мобільного застосунку вимогам лабораторної роботи. Використання Kotlin, Android Studio, Gradle та AndroidX дозволило створити сучасний навчальний прототип мобільної частини системи LifeBalance.
6 КОНТРОЛЬНІ ЗАПИТАННЯ
1. Яка перевага використання Kotlin для Android у порівнянні з Java? Kotlin має більш лаконічний синтаксис, підтримує null-safety та сучасні засоби асинхронного програмування, що знижує кількість типових помилок у мобільній розробці.
2. Навіщо у звіті потрібна Component Diagram? UML діаграма компонентів дозволяє показати структуру мобільного застосунку, залежності між його модулями та взаємодію із зовнішніми сервісами.
3. Як реалізовано зв’язок мобільного застосунку з іншими частинами системи? Мобільний застосунок взаємодіє із серверною частиною через REST API. Сервер зберігає та обробляє дані користувача, а мобільний клієнт відображає їх у зручному інтерфейсі.
